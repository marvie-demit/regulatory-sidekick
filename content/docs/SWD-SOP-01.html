1 Purpose & Scope

This procedure defines the software development lifecycle so medical device software is developed, verified and released safely and to its requirements, using an agile approach.

Scope: all medical device software (SaMD and embedded). It satisfies IEC 62304 and IEC 82304-1, integrates security (IEC 81001-5-1, SEC-SOP-01), AI (AIM-SOP-01) and risk (RSK-SOP-01), and applies agile practices per AAMI TIR45. Design control runs jointly with DEV-SOP-03.

3 Procedure

Guidance (delete before release): Each step is a structured configuration (IDEF0/ICOM): Input, Activity, Responsible (RACI), Method/Criteria (Control), Tools/Template (Mechanism), Output/Record, Clause.

3.1 Software planning & safety classification

InputIntended use; D&D plan; architecture concept
Activity• Prepare the software development plan and the configuration-management plan.
• Assign the IEC 62304 software safety class (A/B/C) and record the rationale.
• Plan the agile increments, the 'definition of done' and the toolchain.
ResponsibleSoftware Lead (R); Risk Manager (Support)
Method / CriteriaIEC 62304 §5.1/§4.3; agile (TIR45)
Tools / TemplateSWD-TPL-01; SWD-TPL-02; SWD-FOR-06
Output / RecordSoftware plan; safety class
ClauseIEC 62304 §5.1/4.3

3.2 Software requirements

InputStakeholder requirements; intended use; risk
Activity• Define software requirements (functional, performance, interfaces, security, usability, data) within agile stories.
• Mark risk-related requirements and ensure each is verifiable.
• Trace requirements to product requirements and review them.
ResponsibleDevelopers (R)
Method / CriteriaIEC 62304 §5.2; ISO/IEC/IEEE 29148
Tools / TemplateSWD-TPL-03 SRS; SWD-FOR-01
Output / RecordSoftware Requirements Specification
ClauseIEC 62304 §5.2

3.3 Architecture & design

InputSRS
Activity• Define the software architecture and detailed design.
• Identify SOUP/third-party components and record them in the SBOM.
• Design segregation for safety/security-critical functions.
ResponsibleDevelopers (R); Security Officer (Support)
Method / CriteriaIEC 62304 §5.3/§5.4; SOUP §8.1.2
Tools / TemplateSWD-TPL-04; SWD-FOR-02; SEC-LIS-01
Output / RecordArchitecture; SOUP list
ClauseIEC 62304 §5.3/8.1.2

3.4 Implementation

InputArchitecture & design
Activity• Implement software in agile increments meeting the 'definition of done'.
• Perform peer code review against the coding standard and security checks.
• Use continuous integration.
ResponsibleDevelopers (R)
Method / CriteriaIEC 62304 §5.5; code review; CI (TIR45)
Tools / TemplateSWD-FOR-04 Code Review
Output / RecordSource code; review records
ClauseIEC 62304 §5.5

3.5 Verification (integration & system test)

InputImplemented software
Activity• Verify software through unit, integration and system testing.
• Run regression testing each increment.
• Record results and maintain test-to-requirement traceability.
ResponsibleDevelopers (R)
Method / CriteriaIEC 62304 §5.6/5.7; regression each increment
Tools / TemplateSWD-TPL-05; SWD-TPL-11/12
Output / RecordVerification records
ClauseIEC 62304 §5.6/5.7

3.6 Validation

InputVerified software
Activity• Validate the software against intended use under representative conditions.
• Include usability validation (HUF) and, for AI, model performance (AIM-TPL-03).
ResponsibleSoftware Lead (R)
Method / CriteriaValidation vs intended use
Tools / TemplateSWD-TPL-13/14; HUF; AIM-TPL-03
Output / RecordValidation records
ClauseISO 13485 7.3.7

InputValidated software
Activity• Confirm all planned activities are complete and known anomalies are acceptable.

ResponsibleSoftware Lead (R)
